# Service Request Catalogue
### Richmond Solutions Inc. — IT Service Management

**Document Version:** 1.1  
**Last Updated:** 2026  
**Owner:** IT Support Team  
**ITIL Practice Area:** Service Request Management, Service Desk, Service Level Management

---

### 1. Purpose

The purpose of this catalogue is to define standardized IT service requests available to employees at Richmond Solutions Inc.

The catalogue provides:
- Approved request types
- Fulfilment expectations
- Approval requirements
- Standard delivery procedures
- Service level targets

This ensures requests are fulfilled consistently, efficiently, and with minimal operational disruption.

---

### 2. Scope

This catalogue applies to all Richmond Solutions employees requiring access to IT-supported services, including:
- User accounts
- Microsoft 365 services
- Endpoint hardware
- Collaboration tools
- Software installations
- VPN access
- Shared drive permissions
- Printing services

---

### 3. Definition of a Service Request

A service request is a request from a user or authorized user for something that is part of normal service delivery — not an incident.

Examples include:
- Password resets
- Software installation
- Onboarding requests
- Access changes
- Hardware requests

Service requests are:
- Pre-approved
- Low risk
- Repeatable
- Operational in nature

### 5. Service Request Workflow

### 5.1. Standard Request Workflow

```text
Request Submitted
↓
Service Desk Review
↓
Approval Validation (if required)
↓
Request Fulfilment
↓
User Confirmation
↓
Ticket Closure
```

### 5.2. Approval Workflow

```text
Submit Request (Portal / Email / Phone)
↓
Service Desk logs ticket and confirms receipt (within 1 hour)
↓
Approval obtained (if required)
↓
Fulfilment by Service Desk analyst
↓
Confirmation sent to requester
↓
Ticket closed
```

### 8. Request Prioritization

Service requests are generally lower priority than incidents.

However, requests may be expedited when:
- Executive onboarding is involved
- Business-critical deadlines exist
- Operational disruption may occur without fulfilment

---

### 9. Request Closure Standards

A request is closed only when:
- Fulfilment is completed
- User confirmation is received where applicable
- Documentation is updated
- Access changes are validated

---

### 10. Security & Compliance Considerations

All requests involving access changes must:
- Follow least privilege principles
- Maintain audit traceability
- Include approval evidence where required

Sensitive access requests may require:
- Secondary approval
- IT Manager review
- Security validation
